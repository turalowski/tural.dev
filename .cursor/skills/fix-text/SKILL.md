# Fix Text

Improve clarity and readability while applying this repo's text style rules.
Prefer small, natural edits. Do not change meaning unless the user asks.

## When to apply

- Writing or editing blog posts (`public/blog/posts/**`)
- Editing site copy, titles, excerpts, or UI strings
- User asks to fix, polish, tighten, or improve text

## Tone

Write like someone sharing an opinion from problems they actually hit, not like a teacher or tutorial author.

- Ground claims in lived experience ("I've seen…", "when this broke for us…").
- Avoid lecture framing: no "here's what you need to know", lesson outlines, or step-by-step teaching voice.
- Prefer first-person judgment over generic best-practice advice.
- Stay direct and conversational. Skeptical is fine; preachy is not.

## Style rules

Apply every rule below. Add new rules at the end of this list.

1. **No dashes.** Do not use em dashes (`—`), en dashes (`–`), or spaced hyphen asides (` - `). Rewrite with a period, comma, colon, or parentheses instead.
2. **Straight apostrophes only.** Use `'` (ASCII U+0027). Never use left/right single quotes (U+2018 / U+2019).

## Workflow

1. Read the target text in full.
2. Fix style-rule violations first.
3. Check tone: opinion from experience, not a lesson.
4. Improve clarity, flow, and grammar without inventing facts.
5. Keep the author's voice. Do not make the copy sound generic.
6. Show the improved text (or apply edits in-file when asked).
